# 配音 Skill 文件

**角色**：Voice Actor（配音）
**对应流程**：STEP 4 — 素材生产（TTS 部分）
**Review 节点**：REVIEW-6（素材质量确认，TTS 部分）

---

## 一、角色定义

配音负责把脚本文字转化为有节奏感的音频。在制片厂里，这一步通过 TTS（文字转语音）工具 + 人工审听完成。

配音质量直接影响完播率——语气不对、停顿不自然，观众 3 秒就划走。这是唯一一个**必须人工审听**的步骤，Layer 1 无法替你检查音频质量。

---

## 二、输入

- 脚本 TTS 版（`_TTS.txt`，含停顿标记）
- Channel_Config.md（读取语气要求）
- 触发词：在 STEP 4 执行时，TTS 文件已备好

---

## 三、TTS 工具配置

### 推荐方案：MiniMax

**MiniMax** 是目前中文 TTS 效果最好的选择之一，支持情感控制和多音色。

**配置步骤**：

1. 注册 MiniMax 开放平台：[minimaxi.com](https://www.minimaxi.com)
2. 在控制台创建应用，获取 API Key
3. 选择语音模型：推荐 `speech-02-hd`（高清版本）
4. 选择音色：根据频道风格选择
   - 冷静叙述型：推荐 `male-qn-qingse` 或 `presenter_male`
   - 温和讲解型：推荐 `female-shaonv` 或 `presenter_female`
   - 具体音色列表请参考 MiniMax 官方文档
5. 将 API Key 和音色选择记录在 `Channel_Config.md` 或 `User_Overrides.md`

**基本调用参数**：

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| `model` | `speech-02-hd` | 高清版本，质量优先 |
| `voice_id` | 根据频道风格选择 | 见上方音色推荐 |
| `speed` | 0.9 - 1.1 | 知识讲解偏慢 0.9，历史叙事偏快 1.05 |
| `output_format` | `mp3` | 通用格式 |

### 备选方案

| 工具 | 适合场景 | 优缺点 |
|------|---------|--------|
| **Azure TTS** | 多语言（中英双版） | 英文效果好，中文略机械 |
| **ElevenLabs** | 英文 YouTube 版 | 英文顶级，中文暂不推荐 |
| **真人录制** | Vlog、教学类 | 效果最好，成本最高 |
| **剪映 / CapCut** | 快速出片、初期测试 | 免费，效果一般 |

## 五、核心规则

- TTS 源文件使用 `_TTS.txt`（含停顿标记），主脚本 `.md` 保持干净
- 语气目标：根据 Channel_Config.md 中的语气标准，选择匹配的音色和语速
- 中英文分别生成独立音频文件
- 生成后**必须人工审听**（TTS 质量无法自动检查）

### 停顿标记放置原则

- 停顿时长统一为 `<#0.25#>`，不使用其他时长
- **克制放置**：默认不加停顿，只在必要时加
- **可以加停顿的位置**（二选其一即可）：
  - 重要句末：该句是段落核心结论，需要让听众消化一下
  - 强调节点：数字、反转、核心术语首次出现前后
- **不加停顿的位置**：普通连接句的句末、列举中间、过渡句
- 全篇停顿标记建议控制在 **5 处以内**（60 秒脚本参考值）

### 特殊发音处理

| 情况 | 处理方式 |
|------|---------|
| 外国人名 | 在 TTS 文件中加注音，例：`杰文斯(jié wén sī)` |
| 英文专有名词 | 保留英文原文，TTS 模型通常能正确识别 |
| 中文多音字 | 补充语境提示，或用同义词替换 |
| 数字/年份 | 写成汉字（`一九七零年` 而非 `1970年`）提高准确率 |

## 六、TTS 稿改写原则（信道重新编码）

**底层原理（香农信息论）：** 文档是高带宽信道——可以加粗、加冒号、分层级，读者能回看。TTS 是低带宽有损信道——线性、单次播放，听众没有回看机会。信道带宽降低时，必须提高冗余度，才能保持同等信息传输量。TTS 稿不是"把文档读出来"，而是针对低带宽信道做重新编码。

**三类重新编码操作（必须全部执行）：**

1. **逻辑词显化**：文档里靠标点和格式传递的因果/条件/转折，TTS 必须用词语说出来。
   - 符号一律转连接词：`→` `;` `、` → `然后` `接着` `同时`
   - 破折号转转折词：`——` → `但是` `而是` `也就是说`
   - 平行结构加因果词：`库存积压，占用现金流` → `库存积压了，所以现金流就被占住了`

2. **转折词密度加倍**：口语的节奏靠转折词撑起来，文档里一个转折词能撑一段，TTS 里每1-2句就要有一个节奏词（`但是` `不过` `然而` `反过来` `换句话说` `更准确地说`）。没有转折词的长句，听起来像在背课文。

3. **关键信息重复强化**：反直觉数字、机制名词、核心结论，首次出现后必须换一种说法再说一遍。听众没有"回看"机会，第一遍没听清就丢了。例：`这个比例是3%` → `只有3%，不到三十分之一`。

**生成TTS后，扫全文检查：**
- 有无 `→` `;` `——`？有则必须转换。
- 有没有连续3句以上没有逻辑词/转折词的段落？有则补上。
- 有没有只出现一次、没有重复的关键数字或术语？有则加一句换说法的强化。

## 七、双层 Review 清单

### 🤖 Layer 1 — AI 自检项

- [ ] 中文 MP3 文件存在？
- [ ] 英文 MP3 文件存在（YouTube 版，如需）？
- [ ] TTS 源文件使用的是 `_TTS.txt`（非主脚本 `.md`）？
- [ ] 特殊发音（人名、专有名词）已在 TTS 文件中标注？

*（TTS 语气和停顿质量无法自动检查，全部进入 Layer 2）*

### 👤 Layer 2 — 创作者判断项（必须亲自听）

- 停顿节点自然吗？有没有让你出戏的机械停顿？
- 语气是否符合 Channel_Config 中的语气标准？
- 有没有明显的读音错误（人名 / 地名 / 专有名词）？
- 语速是否舒适？太快听不清、太慢想跳过？

## 八、常见错误

| 错误 | 表现 | 修正 |
|------|------|------|
| TTS 源文件用错 | 用了 `.md` 而非 `_TTS.txt` | 确认使用含停顿标记的 TTS 版 |
| 人名读音错误 | 外国人名被逐字读 | 在 TTS 文件中加注音 |
| 语速不匹配 | 知识类太快，叙事类太慢 | 调整 speed 参数 |
| 只生成中文版 | 忘记英文版 | 检查 Channel_Config 平台设置 |
| 未审听就通过 | 跳过 Layer 2 | TTS 是唯一必须人工审听的步骤 |
| 数字读法错误 | "1970年"读成"一千九百七十年" | 写成"一九七零年" |
