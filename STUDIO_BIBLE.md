**版本**：v1.1

## 零、为什么用影视工业框架

传统电影工业解决了一个和你完全相同的问题：**如何让一群能力各异的人，按顺序生产出质量可控的故事**。

他们用了 100 年时间打磨出角色分工、Greenlight 机制、每日样片 Review、制片手册（Bible）。这套机制的底层逻辑和你做短视频是同一套——只是规模不同。

**Feige 的判断**：你的每一期视频不是独立产品，是宇宙的一部分。工厂的价值不在于某一期多好，在于系统可重复。

**Kennedy 的判断**：工厂的每一道工序，都有可能在不知不觉中稀释故事的核心。每个 Review 节点的真正作用，是保护创作意图不被流程本身磨损。

## 一、两位顾问的视角分工

在使用本手册时，遇到以下类型的决策，调用对应顾问视角：

| 决策类型 | 调用谁 | 核心问题 |
|---------|--------|---------|
| 这期选题值不值得做？ | **Feige** | 它在宇宙里占什么位置？与其他期的关联度？ |
| 这个脚本的情绪核是否成立？ | **Kennedy** | 观众会在哪一秒感到震撼？那一秒在哪里？ |
| 系统/流程/模板要不要改？ | **Feige** | 改了之后可重复性是否更高？ |
| 品牌决策（要不要发、要不要改）？ | **Kennedy** | 这个决策发布后，观众对你的认知会不会变？ |
| 这一步的输出要不要接受？ | **两者都问** | Feige：符不符合规格？Kennedy：故事灵魂还在吗？ |

## 二、制片厂角色全景（9 个岗位）

> 这 9 个角色是通用的，不绑定任何内容风格或频道定位。每个频道的"创作信仰"注入在各自的 Skill 文件里，不在这里。

| # | 角色 | 影视工业原型 | 在制片厂做什么 | AI 替代程度 |
|---|------|------------|--------------|------------|
| 1 | **制片人** Producer | 好莱坞制片人（Greenlight 决策者） | 选题生成、谜题测验、价值判断 | ⭐⭐⭐⭐ |
| 2 | **编剧** Screenwriter | 剧本作家 | 脚本生成、结构搭建、语气调校 | ⭐⭐⭐ |
| 3 | **导演** Director | 执行导演（视觉翻译者） | 把旁白转化为画面意图（画面拆解） | ⭐⭐⭐⭐⭐ |
| 4 | **美术指导** Art Director | 制作设计 Production Designer | 视觉风格定义、时空锚点体系 | ⭐⭐⭐⭐ |
| 5 | **分镜师** Storyboard Artist | 故事板艺术家 | T2I Prompt 生成 + 运动指令 | ⭐⭐⭐⭐⭐ |
| 6 | **配音** Voice Actor | 配音演员 / 旁白 | TTS 标记生成（停顿、语气节点） | ⭐⭐⭐⭐⭐ |
| 7 | **剪辑师** Editor | 剪辑指导 | 时间线方案、动画指令、节奏编排 | ⭐⭐⭐⭐ |
| 8 | **宣发** Distribution | 市场营销 / 发行 | 封面描述、标题、视频简介 | ⭐⭐⭐⭐ |
| 9 | **质量总监** QC Director | 后期监制 / 质量把关 | 红线审核、完整性校验、最终放行 | ⭐⭐⭐⭐⭐ |

**你的角色**：执行制片人（Executive Producer）。每个 Review 节点，你是唯一的决策人。AI 不 Greenlight，你 Greenlight。

## 四、每个步骤详细规范

每个步骤的统一结构：

```
触发词 → AI 执行 → 交付物
                    ↓
              🔍 REVIEW（你的判断）
             /              \
          通过               不通过
           ↓                   ↓
       下一步骤           修改要求 → AI 重做
```

### STEP 1 — 选题开发（Producer）

**触发**：`生成选题`
**执行角色**：Producer
**AI 工作**：根据频道定位生成选题，包含标题、核心矛盾、30秒开头骨架
**交付物**：选题文档（含标题、钩子观点、对比结构、开头骨架）

**🤖 REVIEW-1 Layer 1 — AI 自检**（自动运行）

| 检查项 | 标准 |
|--------|------|
| 标题包含钩子元素 | ≥ 2 个（权威背书 / 具体数字 / 禁忌警告 / 悬念冲突） |
| 核心对比结构完整 | 两个具体事物（非抽象概念），反差清晰 |
| 30秒开头骨架存在 | 逐句可读，非摘要 |
| 谜题测验已执行 | 有测验结果记录 |
| 与选题库无重复 | 对照已有选题列表 |

**👤 REVIEW-1 Layer 2 — 创作者判断**（你来判断）

| 判断项 | 通过信号 | 失败信号 |
|--------|---------|---------|
| **谜题测验感受** | 听到核心问题，你的第一反应是"为什么？！" | 反应是"哦"或"所以呢" |
| **传播直觉** | 你会想主动把这个话题讲给某个具体的人听 | 只觉得"有意思"，不想传播 |
| **Feige：宇宙位置** | 与已有期数互补或衍生，不重复 | 和某一期说的是同一件事 |
| **Kennedy：情绪核** | 能说清楚观众在哪一秒会"脊背发凉" | 说不清楚震撼点在哪里 |

**通过** → 进入 STEP 2
**不通过** → 注明具体失败项，AI 重新生成或修改

---

### STEP 2 — 脚本开发（Screenwriter）

**触发**：`写脚本` / `生成脚本 [选题文件路径]`
**执行角色**：Screenwriter
**AI 工作**：分两阶段执行
- **第一阶段**：生成脚本主文件，经 Layer 1 + Layer 2 确认定稿后保存
- **第二阶段**（触发词：`生成配套文件`）：脚本定稿后，生成 TTS 标记版（中/英）、封面描述、视频简介
**交付物**：脚本主文件（第一阶段）+ TTS 文件 + 封面/简介文件（第二阶段，详见频道专属 Skill 文件）

**🤖 REVIEW-2 Layer 1 — AI 自检**（自动运行）

| 检查项 | 标准 |
|--------|------|
| Hook 有具体入口 | 实物或历史场景，非抽象概念开场 |
| 六大结构模块存在 | Hook / Contrast / Naming / Mechanism / Mapping / Closing 均可识别 |
| Naming 时刻存在 | 有命名词，且有白话解释 |
| 收束类型匹配 | 揭示系统规律 → 认知收束；有操作空间 → 行动收束 |
| TTS 文件格式正确 | 停顿标记只在 `_TTS.txt`，主脚本 `.md` 无停顿标记 |
| 禁忌内容 | 无毒品 / 疫情 / 政治敏感 / 宗教 / 分裂言论 |
| 文件路径正确 | 保存在 `02_Scripts/` 下 |

**👤 REVIEW-2 Layer 2 — 创作者判断**（你来判断）

| 判断项 | 通过信号 | 失败信号 |
|--------|---------|---------|
| **Kennedy：故事感** | 从头看一遍，每隔 30 秒都有一次"啊原来如此" | 信息正确但看完没感觉 |
| **语气** | 冷静旁观，不共情，像在呈现物理事实 | 出现"我们都经历过"类共情句 |
| **震撼点定位** | 能指出脚本里最强的那一句，知道它为什么强 | 说不出哪里最打动人 |
| **Kennedy：品牌一致** | 符合你的频道声音，发出去不会让老粉觉得"不像你" | 语气或案例风格与频道定位偏移 |

**通过** → 进入 STEP 3（Art Director）
**不通过** → 注明具体模块，AI 修改

---

### STEP 3A — 视觉风格定义（Art Director）

**触发**：脚本通过 Review 后自动进入，或 `定义视觉风格`
**执行角色**：Art Director
**AI 工作**：根据脚本内容，定义全片时空坐标、主视觉色调、摄影风格
**交付物**：视觉风格卡（时代、地域、色调、摄影风格声明）

**🤖 REVIEW-3 Layer 1 — AI 自检**（自动运行）

| 检查项 | 标准 |
|--------|------|
| 所有案例段落有时空坐标 | 时代 + 地域均已声明 |
| 无"整体偏暗"等空话 | 色调说明有具体依据 |
| 跨时空跳跃已声明处理原则 | 多个时代的案例有明确的风格切换说明 |

**👤 REVIEW-3 Layer 2 — 创作者判断**（你来判断）

- 这个视觉风格，和你想要呈现的内容情绪是否匹配？
- 有没有哪个案例段落的时代/地域定义让你觉得"不对，应该是……"？

**通过** → 进入 STEP 3B
**不通过** → 注明具体段落，AI 调整

---

### STEP 3B — 画面拆解（Director）

**触发**：`画面拆解`
**执行角色**：Director
**AI 工作**：把脚本旁白拆成画面单元，标注 A（具象）/ B（抽象/概念）类型
**交付物**：画面拆解文件（画面单元列表 + A/B 标注）

**🤖 REVIEW-4 Layer 1 — AI 自检**（自动运行）

| 检查项 | 标准 |
|--------|------|
| 全文覆盖 | 每句旁白都有对应画面单元，无遗漏 |
| A/B 标注完整 | 所有画面单元已标注类型 |
| B 类有视觉化策略 | 无 B 类留白 |
| Hook 段密度 | Hook 段画面单元数量 ÷ Hook 段字数 > 其他段落同比 |
| 枚举项处理 | 枚举型旁白每项独立一个画面单元 |
| 总数控制 | ≤ 70 个（超过 10 分钟视频例外） |

**👤 REVIEW-4 Layer 2 — 创作者判断**（你来判断）

- 有没有哪个画面单元你脑子里立刻能看见画面，但 AI 的描述和你想的完全不一样？
- B 类的视觉化策略，有没有你觉得"这个比喻不对"的？

**通过** → 进入 STEP 3C
**不通过** → 注明具体编号，AI 调整

---

### STEP 3C — 分镜生成（Storyboard Artist）

**触发**：`生成分镜`
**执行角色**：Storyboard Artist
**AI 工作**：把画面单元转化为 T2I Prompt + 运动指令，按原文块组织
**交付物**：分镜文件（Prompt + 运动 + 视觉盲点标注）

**🤖 REVIEW-5 Layer 1 — AI 自检**（自动运行）

| 检查项 | 标准 |
|--------|------|
| Prompt 自给自足 | 逐条扫描，无"同上"/"比上一个"等相对引用 |
| 时空锚点完整 | 每条 Prompt 含时代 + 地域文化 |
| 静态描述 | 无动作动词作主谓（"走入"/"放下"→ 报错） |
| 无中文文字入镜 | Prompt 无描述图片内中文文字内容 |
| Naming 段处理 | 对应 Naming 画面单元的 Prompt，运动字段为"静止" |
| 视觉盲点汇总 | `⚠️ 视觉盲点` 标注已统一列在文件末尾 |

**👤 REVIEW-5 Layer 2 — 创作者判断**（你来判断）

- 快速翻一遍 Prompt 列表，有没有哪张你脑子里看不到画面（说明描述太模糊）？
- Hook 段的图，排列出来节奏感是否够快？
- 视觉盲点列表：每一条，你决定：替换旁白 / 跳过画面 / 用替代图

**通过** → 进入制作决策链
**不通过** → 注明具体 Prompt 编号，AI 修改

**触发**：剪辑方案通过后，或 `生成封面标题`

## 五、Review 机制 — 双层架构

每个步骤的 Review 分两层，性质完全不同：

```
AI 执行 → 交付物
              ↓
    ┌─────────────────────────────┐
    │  🤖 Layer 1：AI 自检 Agent  │  ← 独立 Agent，与执行 Agent 分开
    │  · 对照规则逐条检查          │
    │  · 只查客观可验证的规格项    │
    │  · 产出结构化自检报告        │
    └─────────────────────────────┘
              ↓
    自检报告 + 交付物 → 一起呈现给创作者
              ↓
    ┌─────────────────────────────┐
    │  👤 Layer 2：创作者判断      │  ← 你，唯一的 Greenlight 人
    │  · 只看 Layer 1 没能判断的  │
    │  · 品质感 / 品牌感 / 直觉   │
    │  · 给出明确指令              │
    └─────────────────────────────┘
              ↓
         通过 → 下一步
         不通过 → AI 修改 → 重新 Layer 1 → 重新 Layer 2
```

### 两层的本质区别

| | Layer 1（AI 自检） | Layer 2（创作者判断） |
|--|------------------|---------------------|
| **性质** | 规格合规性 | 品质判断力 |
| **检查内容** | 客观可查的规则：格式、完整性、红线、技术规范 | 主观判断：故事感、品牌感、震撼感、直觉 |
| **能否自动化** | 是 | 否 |
| **对应顾问** | Feige（规格 / 系统） | Kennedy（故事 / 品牌） |
| **失败后怎么办** | AI 自动修复可修复项，其余列出让创作者决定 | 创作者给具体修改指令 |

### Layer 1 自检 Agent 的工作方式

每个步骤执行完毕后，自检 Agent 自动触发（不需要人工指令）。它：
1. 读取该步骤的交付物
2. 逐条对照该步骤 Skill 文件里的「AI 自检项」清单
3. 产出标准格式报告：

```
=== 自检报告 — [步骤名] ===

✅ 通过（X 项）
  · [规则名]：[简短说明]
  ...

❌ 未通过（X 项）—— 已自动修复
  · [规则名]：发现 [具体问题]，已修改为 [修改内容]

⚠️ 未通过（X 项）—— 需创作者决定
  · [规则名]：发现 [具体问题]，无法自动修复，请指示

📋 建议：[是否可以进入 Layer 2 Review]
```

### Layer 2 创作者的工作方式

收到自检报告 + 交付物后，创作者只需要回答：
- 可修复的问题由 AI 已自动修复，确认或推翻修复结果
- ⚠️ 标记的问题给出具体指令
- 对「人工判断项」清单做出判断（只有这些项需要你的品味）

**你不需要背任何规则。规则是 Layer 1 的工作。你只需要判断。**

### Review 失败后的处理

| 失败类型 | 处理方式 |
|---------|---------|
| 规格问题（格式错误、遗漏项） | Layer 1 自动修复，修复后重新自检 |
| 结构性问题（Hook 缺失、逻辑断裂） | 退回当前步骤，AI 重做对应模块 |
| 红线问题（内容违规） | 退回 STEP 2，替换案例 |
| 品牌问题（风格偏移） | Kennedy 视角优先，宁可慢也不发 |

### 打包交付的意义

这套双层架构就是产品本身：
- **创作者（你的学员）** 只需要做 Layer 2——用判断力，不需要背规则手册
- **Layer 1** 帮助创作者在上手初期不犯低级错误，等熟练后可以跳过或简化
- **系统可交付**：换一套 Skill 文件，同一套 Review 架构适配任何频道类型

## 六、通用性设计 — 如何适配不同频道类型

本 Studio Bible 不包含任何频道专属的创作信仰。这些应注入在各角色的 Skill 文件里：

```
STUDIO_BIBLE.md          ← 你正在读这个（通用框架，不动）
    ↕
01_Producer/Producer_Skill.md    ← 频道专属：选题标准、谜题测验、酒桌原则
02_Screenwriter/Screenwriter_Skill.md  ← 频道专属：爆款三要素、结构模板、语气要求
...
```

**换一个频道 = 换一套 Skill 文件，Studio Bible 不变。**

不同内容类型的角色权重参考：

| 内容类型 | 重度角色 | 轻度角色 |
|---------|---------|---------|
| 知识讲解 / 历史/人物故事 | 编剧 + 导演 + 分镜师 + 美术 | 配音用TTS即可 |
| 商业故事 | 编剧 + 导演 + 分镜师 | 配音用TTS即可 |
| 创意叙事 | 编剧（核心）+ 分镜师 + 美术 | 配音用TTS即可 |
| 睡前故事 | 编剧 + 配音（声音是核心产品）| 美术轻度，画面极简 |
